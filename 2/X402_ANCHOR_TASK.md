# X402 锚点服务与后端优化任务

## 1. 目标
利用 X402 的链上存证能力，增强 B2P 仲裁结果的公信力，并优化后端性能。

## 🏗️ 项目架构背景（必读）
*   **锚点服务：** 现有 `backend/src/anchor/github_anchor.py` 负责将交易哈希 Push 到 GitHub。
*   **存证逻辑：** 我们的信誉系统是分布式的，GitHub 是主要的“公共账本”。X402 链上数据应作为 GitHub Commit 的补充验证层。
*   **异步处理：** 后端使用简单的线程或 asyncio 处理耗时任务，X402 的链上写入必须异步化，避免阻塞 API 响应。

## 2. 核心任务分解

### 第一步：仲裁结果链上锚定
- [ ] 在 `backend/src/anchor/github_anchor.py` 中增加 X402 锚定逻辑。
- [ ] 逻辑：每次 Push 仲裁裁决到 GitHub 后，提取 Commit Hash 并写入 X402 链上。
- [ ] 目的：防止 GitHub 仓库被篡改，确保证据链永久有效。

### 第二步：批量结算与性能优化
- [ ] 研究 X402 的 Batch Settlement（批量结算）方案。
- [ ] 设计后端队列：将高频小额交易的 X402 支付请求合并处理，降低 Gas 成本。
- [ ] 优化 `server.py` 中的异步任务处理，避免 X402 API 调用阻塞主线程。

### 第三步：多链容灾配置
- [ ] 配置备用支付通道（如从 Base 切换到 Arbitrum）。
- [ ] 实现 X402 中继节点宕机时的自动降级策略。

## 3. 预期成果
*   **不可篡改存证：** 仲裁记录同时存在于 GitHub 和 X402 支持的公链上。
*   **低成本运行：** 通过批量结算显著降低高频微交易的运营成本。
*   **高可用性：** 即使单一区块链网络拥堵，交易系统仍能正常运行。
